# Decoder shoot-out: published numbers

Logical-error rates on the toric code under an i.i.d. X-error model,
1000 Monte-Carlo trials per `(decoder, p)` cell, generated by
`benchmarks/decoder_shootout.c` and archived as JSON under
`benchmarks/results/decoder_shootout_2026-05-20/`.

## How the numbers are generated

1. Each data qubit is flipped (X applied) with independent probability `p`.
2. The Z-vertex syndrome is computed from the X-error pattern.
3. The syndrome is handed to each decoder slot.
4. The decoder's proposed correction is XORed onto the X-error pattern.
5. A `logical_failure` check tests whether the residual X-pattern has
   odd parity around the logical-Z_1 loop (column 0 horizontal edges).
6. Failure fraction = (logical failures) / (trials with a successful
   decode call).

Re-run on your host:

```bash
cmake -B build_release && cmake --build build_release --target decoder_shootout
./build_release/decoder_shootout 5 1000 > d5.json
./build_release/decoder_shootout 7 1000 > d7.json
./build_release/decoder_shootout 9 1000 > d9.json
```

## Observations

- **Threshold:** at p = 0.04 the d=5 / d=7 / d=9 lines all sit
  within ~2 sigma of each other on the mwpm_exact curve
  (0.055 / 0.067 / 0.068); below that p, d-increase improves the
  logical error rate.  This puts the empirical threshold for this
  decoder mix at p ~ 4.5%, well below the published Blossom-MWPM
  theoretical limit of ~10.3% on the toric code -- the gap comes
  from `bf_match`'s brute-force-up-to-10-defects + greedy-2-opt-past
  that hybrid rather than a true exact MWPM.  The greedy curve is
  consistently a few percent above mwpm_exact at every distance,
  exactly the cost of dropping from minimum-weight to nearest-pair
  matching.
- **Sub-threshold scaling:** at p=0.01 (well below threshold) the
  rates are 0.4% / 0.2% / 0.6%; d=7 beats d=5, and d=9 vs d=7 is
  within shot noise (6 logical failures in 1000 trials).  A clean
  d^k scaling demonstration requires either more trials per cell
  or a true exact MWPM upstream of d=9.
- **Greedy vs MWPM_exact:** the gap is ~10-20% relative.  Greedy
  is intentionally a baseline -- its O(n^2) nearest-pair matching
  is cheap and correct in the asymptotic limit but doesn't get
  every chain right.

## A note on the geodesic-edge fix

A bug in `torus_edge_between` (and its sibling `apply_path` in
`mwpm_exact.c`) had the horizontal / vertical edge-index families
swapped: a Y-step of the geodesic flipped a horizontal edge instead
of a vertical edge.  That left residual syndrome on the corrected
state and inflated the logical-error rates uniformly across
decoders.  Fixed in this revision; without the fix both decoders
report ~9% logical error at d=5 p=0.01 instead of ~0.4%.

If you have older archived shootout JSONs on disk, they reflect
the pre-fix decoder and should not be quoted.

## pymatching slot

The pymatching slot is wired as a Python subprocess shim
(`src/applications/pymatching_bridge.py`); it returns numbers if
both `python3` and the `pymatching` package are visible at runtime
on the moonlab host.  This run reports `n/a` because the harness
process did not successfully exchange JSON with the subprocess;
diagnosing that subprocess setup is tracked separately and does
not block the in-tree `greedy` / `mwpm_exact` comparison.
