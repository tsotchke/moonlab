Audit cut on 2026-04-19 against master @ b044317 (post-P5.08-closure).
Leads with what's broken; there is no "wins" section.

## Status

**Blocks release.**  Twelve items fall into four categories:

## 1. Memory safety (C core)

All six were NULL-deref or pointer-corruption patterns on the OOM path.
Not user-reachable under normal load, but "public release" means
strangers on random hardware, and an OOM on a 4 GB laptop should not
segfault.

| # | File:line                      | Pattern                         | Fix commit |
|---|---------------------------------|---------------------------------|------------|
| 1 | `src/quantum/measurement.c:287` | `calloc` used without NULL check before `outcome_probs[outcome]` | this audit |
| 2 | `src/quantum/measurement.c:479` | `malloc` used without NULL check before `cdf[i]`              | this audit |
| 3 | `src/algorithms/chemistry/chemistry.c:203` | `realloc` return assigned to `h->h1` after `num_h1++` — if realloc fails, `num_h1` is now wrong | this audit |
| 4 | `src/algorithms/chemistry/chemistry.c:215` | same pattern for `h->h2`                                      | this audit |
| 5 | `src/algorithms/chemistry/chemistry.c:254` | JW transform realloc without NULL check; loop would deref NULL on OOM | this audit |
| 6 | `src/algorithms/chemistry/chemistry.c:281` | same pattern for the two-electron JW path                     | this audit |

Fix pattern was uniform: grow via a temporary, bail on NULL, commit
the count only after the realloc succeeds.  Trivial semantics change,
big correctness improvement.

## Python bindings open issues (items 8, 9) — CLOSED

Both resolved in a follow-up commit.  Summary of the work:

- New non-inline C helpers `quantum_entropy_ctx_create_hw` and
  `quantum_entropy_ctx_destroy` in `src/utils/quantum_entropy.{c,h}`.
  These allocate a hardware-backed `quantum_entropy_ctx_t` using the
  existing `entropy_init` / `entropy_free` surface so ctypes-based
  bindings (Python, Rust) can build a working context without
  seeing the `static inline` `quantum_entropy_init`.
- `bindings/python/moonlab/core.py` constructs a process-wide
  `_DEFAULT_ENTROPY_CTX` at import time and threads it through
  `quantum_measure_all_fast`.  `atexit` releases it at shutdown.
- `bindings/python/moonlab/algorithms.py` was leaking four distinct
  struct-layout mismatches against the C side
  (`CQAOAResult`, `CVQEResult`, `CGroverResult`,
  `CBellTestResult`, `CGroverConfig`) plus two missing entropy-ctx
  argtypes (`vqe_solver_create`, `qaoa_solver_create`) and one
  argument-order bug (`bell_test_chsh`).  All fixed by mirroring the
  C `typedef struct` layouts byte-for-byte and passing
  `_DEFAULT_ENTROPY_CTX` into the create-calls.  `solve_maxcut`
  additionally converts the Ising energy back to MaxCut cost via
  `(total_weight - best_energy) / 2`.
- `bindings/python/tests/test_algorithms.py` module-level skip
  removed; 33/34 tests now pass.  One test
  (`test_chsh_product_state_classical`) is marked `xfail(strict=True)`
  pending a separate fix of a pre-existing bug in
  `src/algorithms/bell_tests.c` that returns S ~ 2.8 for the
  separable |++> state (analytic value with standard CHSH angles is
  0).  That bug is tracked in
  `project_broken_subsystems_v012.md` as a new entry.
- Two previously-skipped `test_quantum_state.py::TestMeasurement`
  cases are unskipped and now exercise the entropy-ctx path.
- New ctest target `python_bindings_pytest` runs the pytest suite
  on every build (gated on pytest being importable).  56/56 ctest
  green locally.

## New SEV-2 finding to track

**bell_test_chsh returns ~2.828 regardless of input being separable.**
`test_chsh_product_state_classical` on |++> (product state) gives
S = 2.856 ± statistical noise, where the analytic value is 0 and
the Bell inequality requires S <= 2 for any LHV theory.  Symptom
consistent with the correlator computation ignoring or
mis-rotating the input state; the entangled |Phi+> case gives
expected 2.828, so the function does use `state`, but not
correctly for non-Bell inputs.  Added to
`memory/project_broken_subsystems_v012.md`.

## What this audit does NOT cover

- Deep static analysis (clang-tidy, Coverity).  55/55 ctest green and
  the diagnostics stream from the language-server rebuild are clean
  of new warnings, but that is not a substitute for a proper sweep.
- Fuzzing / ASAN CI.  The sanitizer-build is already wired in
  (`QSIM_ENABLE_SANITIZERS=ON`) but does not run as a tier in the
  default ctest invocation — a Tier-C sanitiser pass should run
  before the tag.
- MPI correctness at >2 ranks.  Still partially covered; the
  existing broken-subsystems memory note stands.
- GPU backends other than Metal + Eshkol.  Not exercised on CI.
