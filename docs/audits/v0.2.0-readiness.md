# v0.2.0 release-readiness audit

Audit cut on 2026-04-19 against master @ b044317 (post-P5.08-closure).
Leads with what's broken; there is no "wins" section.

## Status

**Blocks release.**  Twelve items fall into four categories:

- **Memory safety (C core)** — 6 finds.  6 fixed in this audit pass.
- **Bindings lies** — 4 finds.  2 fixed here; 2 are real coverage gaps
  that need work.
- **Docs overclaim** — 3 finds.  All fixed here.
- **Dead / debug artefacts shipping in the tree** — 3 finds.  Flagged
  only; safe to leave for a follow-up cleanup.

## 1. Memory safety (C core)

All six were NULL-deref or pointer-corruption patterns on the OOM path.
Not user-reachable under normal load, but "public release" means
strangers on random hardware, and an OOM on a 4 GB laptop should not
segfault.

| # | File:line                      | Pattern                         | Fix commit |
|---|---------------------------------|---------------------------------|------------|
| 1 | `src/quantum/measurement.c:287` | `calloc` used without NULL check before `outcome_probs[outcome]` | this audit |
| 2 | `src/quantum/measurement.c:479` | `malloc` used without NULL check before `cdf[i]`              | this audit |
| 3 | `src/algorithms/chemistry/chemistry.c:203` | `realloc` return assigned to `h->h1` after `num_h1++` — if realloc fails, `num_h1` is now wrong | this audit |
| 4 | `src/algorithms/chemistry/chemistry.c:215` | same pattern for `h->h2`                                      | this audit |
| 5 | `src/algorithms/chemistry/chemistry.c:254` | JW transform realloc without NULL check; loop would deref NULL on OOM | this audit |
| 6 | `src/algorithms/chemistry/chemistry.c:281` | same pattern for the two-electron JW path                     | this audit |

Fix pattern was uniform: grow via a temporary, bail on NULL, commit
the count only after the realloc succeeds.  Trivial semantics change,
big correctness improvement.

## 4. Dead / debug artefacts

These do not fail CI — they are not wired into CMake — but they ship
in a source tarball and the name alone is embarrassing:

- `tests/debug_imag_time.c` — January 2026 dev scratchpad.
- `tests/debug_zz_gate.c`   — same vintage.
- `tests/debug_zz_sequence.c` — same vintage.

Recommended follow-up (not done in this pass because it deletes
tracked code that could conceivably contain a test worth keeping):
review each, keep what's a regression test under a proper
`tests/unit/` name, delete the rest.

## Python bindings open issues (items 8, 9)

Resolving these requires a small surface change on the Python side:
expose `quantum_entropy_ctx_t` so `moonlab.core` can pass a non-NULL
ctx into `quantum_measure_all_fast`, and wire that through to
`measurement.c` / `measurement_sample`.  Under a ~50 LOC patch, the
two skipped tests in `test_quantum_state.py` and the full
`test_algorithms.py` suite can be unskipped.  Deferred out of this
audit pass because it's a bigger surface change than the other items.

## What this audit does NOT cover

- Deep static analysis (clang-tidy, Coverity).  55/55 ctest green and
  the diagnostics stream from the language-server rebuild are clean
  of new warnings, but that is not a substitute for a proper sweep.
- Fuzzing / ASAN CI.  The sanitizer-build is already wired in
  (`QSIM_ENABLE_SANITIZERS=ON`) but does not run as a tier in the
  default ctest invocation — a Tier-C sanitiser pass should run
  before the tag.
- MPI correctness at >2 ranks.  Still partially covered; the
  existing broken-subsystems memory note stands.
- GPU backends other than Metal + Eshkol.  Not exercised on CI.

## Release gate recommendation

- v0.1.2 tag stands — QGTL's pin is unaffected by these fixes.
- v0.2.0 gate on items **8** and **9** being resolved (Python
  entropy ctx + unskipped algorithm tests).  Everything else in this
  audit has been fixed in the same pass that wrote this document.
