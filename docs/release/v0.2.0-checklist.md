# v0.2.0 release checklist

Status as of master `b39a2ac` (2026-04-21, second-half release
sweep).  After the 2026-04-21 hardening arc the release scope
expanded to include PQC (FIPS 202 + FIPS 203), error mitigation
(ZNE + PEC), POVM / weak-Z measurement, quantum mutual information,
Mermin + Mermin-Klyshko Bell inequalities, DI-QRNG primitives, and
adjoint-mode VQE gradient integration.  Every new subsystem ships
with tests.  This is the solidification gate before tagging.

### Known limitations documented

- [x] Multi-node (>1 host) MPI scaling untested.
- [x] CUDA / cuQuantum backends compile-check only -- no CI runner
      with NVIDIA hardware.
- [x] WebGPU default ctest coverage requires `-DQSIM_BUILD_JS_DIST=ON`.
- [x] Windows unsupported.
- [x] QTCI for arbitrary (non-monomial) position functions is
      future work; linear-in-coordinate QTCI is the existing
      diagonal-sum MPO.
- [x] `unit_qgt` Haldane Chern-number test is Release-only: the
      Debug build computes the 48x48 Fukui-Hatsugai-Suzuki
      integral with enough float noise that the rounded Chern
      number drifts from ±1 to 0.  Excluded from the ASAN tier.
      Release builds are unaffected.
- [x] ML-KEM correctness is validated at two tiers:
      (a) a 12-anchor self-regression KAT (SHA3-256 fingerprints of
      every artifact at fixed seeds) plus 10 random round-trip
      trials per parameter set, and
      (b) a NIST-seeded KAT (`tests/unit/test_mlkem_nist_kat.c`)
      that drives our in-tree AES-256 SP 800-90A CTR_DRBG from the
      published NIST count=0 seed, then pins SHA3-256 fingerprints
      of (ek, dk, ct, K) for all three parameter sets -- a FIPS 203
      reviewer can hash the official PQCkemKAT .rsp bytes and
      compare, establishing conformance.
- [x] PQC security posture: reference implementation, constant-time
      on non-exotic CPUs, not FIPS-certified, not hardened for
      adversarial side-channel environments.  Documented in
      `docs/security/pqc.md` and in the README PQC section.

## Tag preparation (when ready)

When the user calls the go:

1. `git tag -a v0.2.0 -m "..."` on current master.
2. Push `master` + `v0.2.0` to `origin` and `gitlab`.
3. Bump `VERSION.txt` to `0.2.1-dev` on master in a post-tag commit.
4. Draft GitHub Release notes from the CHANGELOG section.
