# MoonLab v1.1.0 — native CUDA execution

Released 2026-07-11. Version 1.1.0 brings MoonLab's CUDA state-vector backend
online on Tegra and discrete NVIDIA GPUs while preserving the existing CPU gate
API.

## Highlights

- `quantum_state_create_gpu()` creates a CUDA-backed state. Twenty-one existing
  one- and two-qubit gate operations dispatch transparently when that state is
  used; CPU-created states retain the 1.0 behavior.
- The CUDA backend supports state creation, synchronization, host/device copy,
  probabilities, norms, and native 1q/2q/CNOT/X/H operations using managed
  memory on both Jetson and PCIe GPUs.
- `moonlab_vqe_gradient` joins the stable C ABI, raising the ABI version from
  0.3.0 to 0.4.0. It uses reverse-mode adjoint differentiation for supported
  noise-free ansaetze and analytic parameter shift otherwise.
- Jetson and x86-64 NVIDIA validation cover Bell-state construction, a 20-gate
  CPU/GPU parity circuit, and MPI processes with independent GPU contexts.
- Release distribution now includes native Windows x64 and ARM64 ZIPs, built
  with ClangCL and verified through an external CMake consumer before upload.

## Build CUDA

```bash
cmake -S . -B build-cuda \
  -DCMAKE_BUILD_TYPE=Release \
  -DQSIM_ENABLE_CUDA=ON \
  -DCMAKE_CUDA_ARCHITECTURES=86
cmake --build build-cuda -j
```

`QSIM_ENABLE_CUQUANTUM=ON` additionally requires NVIDIA cuQuantum. MoonLab's
own CUDA backend does not require cuQuantum and is the supported Jetson path.
Non-CUDA builds retain weak fallback symbols and return
`QS_ERROR_NOT_SUPPORTED` when a GPU state is requested.

## Compatibility

- Project version: 1.1.0
- Stable ABI: 0.4.0
- CMake: 3.20 or newer
- CPU archives remain portable within their target architecture because
  release jobs disable `QSIM_NATIVE_ARCH`.

For the complete implementation ledger, validation results, known issues, and
deferred multi-GPU work, see the [1.1.0 changelog](../../CHANGELOG.md#110---2026-07-11).
